# Gestión de procesos

## `docker exec`
Se utiliza para ejecutar un proceso nuevo dentro de un contenedor que ya está en estado running.

- **Naturaleza**: Crea procesos secundarios. Puedes tener infinitos exec corriendo en paralelo en el mismo contenedor.
- **Persistencia**: Cada vez que ejecutas exec, abres una sesión totalmente nueva y aislada de otros exec anteriores. Al salir, el proceso secundario muere, pero el contenedor sigue vivo.
- **Uso común**: `docker exec -it mi_contenedor bash` (Entrar a explorar o ejecutar comandos manuales).

## `docker attach`
Sirve para acoplar tu terminal al proceso principal (PID 1) que se definió originalmente con el comando run.

- **Naturaleza**: No crea nada nuevo, solo te "conecta" a lo que ya está pasando en la consola principal del contenedor.
- **Riesgo**: Si te conectas con attach y presionas Ctrl + C, estarás enviando la señal de parada al proceso principal, por lo que apagarás el contenedor.
- **Salida** segura: Para desconectarte sin apagar el contenedor, usa la secuencia Ctrl + P seguida de Ctrl + Q.
- **Uso común**: `docker attach mi_contenedor` (Para ver logs interactivos o interactuar con el proceso base).

# Arrancar y parar contenedores
Así como podemos crear contenedores con `docker run`, podemos parar un contenedor que está corriendo con el comando `docker stop <nombre-contenedor>`.
También, un contenedor que está parado podemos volver a arrancarlo con `docker start <nombre-contenedor>`.

> [!IMPORTANT] Importante
> Un contenedor que necesita entradas de teclado (como una shell) al crearlo sin el modo interactivo, el contenedor no se mantendrá levantado al querer ejecutar `docker start`.
> Debemos crearlo en modo interactivo (`-it`) para que cuando lo levantemos se quede corriendo de fondo.

Cuando un contenedor está corriendo, podemos "meternos" al contenedor con el comando `docker attach <nombre-contenedor>`. Esto vincula nuestra sesión de terminal al contenedor. Esto hace que si el contenedor está corriendo un comando y nos metemos al contenedor, veremos el comando que se estaba ejecutando.
Notar que al ejecutar `exit` el contenedor se para. Para salir de un contenedor sin pararlo, podemos usar la secuencia de escape de Docker (`Ctrl + P`) y luego presionar la letra `Q`. Es decir, `Ctrl + P + Q`.

## Alternativa: `docker kill`
Otra forma de parar un contenedor es usando el comando `docker kill`.
Analizandolo con `docker stop`, ambos detienen un contenedor, pero la principal diferencia entre ambos comandos es la cortesía con la que se pide al contenedor que se detenga.

### `docker stop`
El cierre *formal*. Docker le envía una señal `SIGTERM` (Signal Terminate) al proceso principal del contenedor, teniendo un tiempo de gracia (por defecto de 10 segundos) en donde el contenedor realiza todas las tareas para que se detenga de forma correcta.
Si después de esos 10 segundos el contenedor no se detiene por completo, se realiza un `docker kill` automáticamente.

> [!TIP] Dato útil
> Si tenemos una aplicación pesada que no llega a detenerse en 10 segundos, podemos dar un margen mayor de tiempo con el comando `docker stop -t 30 <nombre-contenedor>`.
> En este ejemplo, le da 30 segundos de tiempo de gracia.

### `docker kill`
El *corte de energía*. Docker envía una señal `SIGKILL`. En este caso, el proceso no hace absolutamente nada, el kernel de nuestro sistema operativo anfitrión fuerza la detención del proceso principal, sin importar el estado en el que esté el contenedor. Es el equivalente a desenchufar una computadora de la pared mientras está encendida.

# Borrar imágenes y contenedores

## Contenedores
Al ver los contenedores que tenemos creados en nuestro sistema con `docker ps -a`, podemos eliminar contenedores con el comando `docker rm <ID_CONTENEDOR>`. Alternativamente podemos usar el nombre del contenedor.
Esto nos devolverá el ID del contenedor que acabamos de borrar, y al ejecutar `docker ps -a`, veremos que el contenedor se ha borrado.

```bash
$ docker ps -a
CONTAINER ID   IMAGE                  COMMAND                  CREATED              STATUS                        PORTS     NAMES
41387db82e0d   ubuntu:noble           "/bin/bash"              About a minute ago   Exited (137) 27 seconds ago             ubuntu-noble

$ docker rm 41387db82e0d
41387db82e0d

$ docker ps -a
CONTAINER    ID   IMAGE    COMMAND    CREATED    STATUS    PORTS    NAMES
```

## Imágenes
Luego de ver las imágenes descargadas en nuestro sistema con `docker images`, podemos borrarlas con el comando `docker rmi <ID_IMAGEN>`, o alternativamente usando el nombre de la imagen completo con su tag `docker rmi <NOMBRE_IMAGEN>:<TAG>`. Ésta última es la recomendada para evitar borrados accidentales cuando hay varias imagenes con distintos tags pero a su vez tienen el mismo ID.

> Notar que no podemos borrar una imagen que está siendo usada para un contenedor en nuestro sistema.
> Se puede forzar el borrado (`-f`) o borrar el contenedor y luego borrar la imagen.

```bash
$ docker images
                                                                                                           i Info →   U  In Use
IMAGE                  ID             DISK USAGE   CONTENT SIZE   EXTRA
ubuntu:noble           c4a8d5503dfb        119MB         31.7MB    U

$ docker rmi c4a8d5503dfb # O en su lugar: docker rmi ubuntu:noble
Untagged: ubuntu:noble
```

Cuando borres una imagen que es única (no comparte capas ni nombres con otras) también se verá algo así:

```bash
Untagged: my-app:latest
Untagged: my-app@sha256:cbbf2f9a...
Deleted: sha256:f603c467...
Deleted: sha256:a12b3c4d...
Deleted: sha256:d5e6f7g8...
```
