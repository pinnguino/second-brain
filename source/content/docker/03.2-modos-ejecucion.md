# Modos de ejecución
Algunos contenedores, como el de Ubuntu, al correrlos con `docker run ubuntu` se paran inmediatamente. Esto se debe a que simplemente el contenedor no tiene nada que hacer. Pero esto viene a cuenta para mencionar que podemos correr un contenedor en varios modos, que hacen que estos contenedores y otros más puedan ser usados según las necesidades del administrador.

## Modo interactivo
Para el modo interactivo, podemos usar las opciones `-it`.
Para entender este modo un poco mejor, debemos mencionar de forma rápida lo siguiente sobre los procesos en Linux.
En Linux, los procesos tienen tres canales de comunicación estándar:

- `stdin`: Por donde el proceso recibe datos (usualmente desde el teclado).
- `stdout`: Por donde el proceso imprime información (la pantalla).
- `stderr`: Por donde el proceso envía mensajes de error.

**Opción `-i`**: Esto hace que Docker simplemente inicia el contenedor y la sesión de terminal en donde iniciaste el contenedor se queda "*escuchando*" lo que el contenedor tenga que decir (`stdout`), pero ignora las entradas del teclado.
**Opción `-t`**: Le asigna una pseudo-terminal al contenedor. En esencia, "*engaña*" al proceso dentro del contenedor para que crea que está conectado a una consola real de Linux.

Normalmente se usa la opción `-t` junto con la opción `-i`. Si estás interesado en profundizar en el uso de estas dos opciones, la documentación explica más en detalle [estas opciones](https://docs.docker.com/reference/cli/docker/container/run).

Si usamos estas opciones con una imagen de Ubuntu levantaremos un contenedor y nos meteremos dentro del mismo en la misma terminal, donde todos los comandos/acciones que realice **sólo afectarán al contenedor**.
Al salir de la sesión de el contenedor de Ubuntu (con el comando `exit`) volvemos a la sesión original de nuestro sistema principal y el contenedor se para.

## Modo background
La otra forma de arrancar un contenedor es en el modo background (*detached*).
En este modo permite que un contenedor se pueda correr en segundo plano sin que tome el control de la sesión de terminal que inicia el contenedor.
Este modo es particularmente útil para contenedores que tienen procesos a modo de servicios o daemons, como `nginx`, `postgres`, `tomcat`, entre otros.
Para poder crear un contenedor de esta forma simplemente ejecutamos `docker run -d <nombre-contenedor>`.
Veámoslo con un ejemplo con la imagen de PostgreSQL:

```bash
$ docker run --name mi-postgres -e POSTGRES_PASSWORD=clavesecreta -d postgres
Unable to find image 'postgres:latest' locally
latest: Pulling from library/postgres
Digest: sha256:78481659c47e862334611ccdaf7c369c986b3046da9857112f3b309114a65fb4
Status: Downloaded newer image for postgres:latest
f4d016d2131132193c39e4a0f6b5f4e00f11e2c6d3bb5cf19b204069197a79f0
```

Aquí, vemos que usamos varias flags para crear este contenedor, veámosla paso a paso:

- `--name`: Le asigna un nombre al contenedor, evitando que se cree con un nombre aleatorio.
- `-e`: Usado para configurar variables de entorno. En este caso, la contraseña del superusuario.
- `-d`: Como es un servicio, le indicamos que este contenedor se debe correr en segundo plano.

## Modos: Interactivo vs Background
**Modo interactivo**: La terminal "*se mete*" dentro del contenedor. Si la imagen es Ubuntu, por defecto te dará acceso a una shell (`bash`). Todo lo que se escriba se ejecuta ahí dentro, y al cerrar la sesión con `exit`, <u>el proceso principal termina y el contenedor se detiene</u>.
En el modo interactivo se pueden ver más rápido los logs: Se pueden ver los errores y mensajes en tiempo real.

**Modo detached**: Cuando usamos `docker run -d ubuntu`, le estamos diciendo a Docker: "Corre esto en segundo plano y devuélveme el ID del contenedor"
El problema: Al querer correr un contenedor de Ubuntu pasa algo curioso: *el contenedor se muere casi al instante*. Esto sucede porque un contenedor vive mientras su proceso principal esté activo. Como la imagen de Ubuntu no tiene un servicio (como PostgreSQL) corriendo de fondo, sino que espera una interacción humana (`bash`), al no haber nadie "escribiendo", el proceso termina y el contenedor se apaga.
En el modo detached, para ver los logs debemos ejecutar `docker logs <ID>` para ver qué pasó.

**Conclusión**: Los modos en los que se pueden inciar un contenedor dependen del tipo de imagen. <u>Revisar la documentación de la imagen en Docker Hub</u> para ver las capacidades de ejecución en contenedores.
