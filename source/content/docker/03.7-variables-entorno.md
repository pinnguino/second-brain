# Variables de entorno
Las variables de entorno son pares de `CLAVE=VALOR` que residen en el entorno del sistema operativo del contenedor. En el desarrollo backend, son esenciales para manejar datos sensibles (como credenciales de bases de datos) o configuraciones que cambian según el entorno (desarrollo, testing, producción).

## 1. ¿Por qué usarlas?

* **Seguridad**: Evita "hardcodear" contraseñas o tokens en el código o en el `Dockerfile`.
* **Portabilidad**: Una misma imagen puede comportarse de forma distinta (conectarse a diferentes DBs) solo cambiando la variable (*contenedores dinámicos*).
* **Flexibilidad**: Permiten configurar parámetros de imágenes oficiales, como la contraseña de una base de datos.

## 2. Cómo pasar variables con el comando `run`

La forma más directa es mediante la opción `-e` o `--env`.

### Sintaxis básica

```bash
$ docker run -e CLAVE=valor <imagen>
```

Notar que el nombre de las *variables son case-sensitive*.
Luego, para verificar si las variables de entorno llegaron correctamente, dentro del contenedor podemos ejecutar el comando `env`, que nos devuelve todas las variables de entorno del sistema.

> [!NOTE] Nota
> Por convención general y buenas prácticas se recomienta que la clave se escriba con mayúsculas.
> Sin embargo, no es obligatorio.

### Ejemplo práctico con Postgres

En este ejemplo, según la documentación oficial, para que un contenedor de Postgres inicie correctamente, es obligatorio definir la clave del superusuario:

```bash
$ docker run --name mi-postgres -e POSTGRES_PASSWORD=mi_clave_segura -d postgres
```

## 3. Uso de archivos `.env` (Best Practice)

Cuando tienes muchas variables (por ejemplo, para una API en **FastAPI** que requiere puerto, base de datos, claves de API y modo debug), pasarlas una por una en la terminal es ineficiente.

Para solucionar esto, se puede crear un archivo de texto plano llamado `.env`:

```bash
# Archivo .env
DATABASE_URL=postgres://user:pass@localhost:5432/mydb
DEBUG_MODE=true
API_KEY=12345abcde
```

Y luego cargar todas las variables juntas usando `--env-file`:

```bash
$ docker run --env-file .env mi-app-backend
```

## 4. Diferencia entre `ENV` (Dockerfile) y `-e` (Runtime)

Es importante no confundir dónde se definen:

* **Instrucción `ENV` en Dockerfile**: Define valores por defecto que se graban en la imagen. Son útiles para rutas de instalación o versiones de software.
* **Opción `-e` en el comando `run`**: Sobrescribe cualquier valor definido en el `Dockerfile`. Es el método que usamos para configurar la instancia específica que estamos creando.

## 5. Visualización de variables en un contenedor

Si queremos verificar qué variables tiene un contenedor que ya está corriendo, se puede usar:

1. **`docker inspect`**: Busca la sección `"Env"`.
2. **`docker exec`**: Ejecuta el comando `env` dentro del contenedor:
```bash
$ docker exec mi-contenedor env
```

> [!WARNING] Seguridad
> Las variables pasadas con `-e` son visibles para cualquier usuario que pueda ejecutar `docker inspect`.
> Para datos extremadamente sensibles en entornos productivos, se recomienda el uso de **Docker Secrets**.
